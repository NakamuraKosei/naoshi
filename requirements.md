## 1. サービス概要

### 1.1 サービス名
**Naoshi（ナオシ）**

### 1.2 一言で言うと
日本の大学生向け、日本語特化のAIテキスト人間化（ヒューマナイザー）Webサービス。

### 1.3 背景
ChatGPT／Claude／Geminiなど生成AIの普及により、大学生がレポート作成にAIを使うことは一般的になった。一方で、AI特有の文体（機械的な接続詞・過剰な整然さ・冗長な言い回し）は教員や同級生に容易に検出され、学生は「AI感を消す」作業に時間を取られている。既存のAIヒューマナイザー（Sidekickerなど）は主に英語圏向けで、日本語の学術レポートに最適化されたサービスは存在しない。

### 1.4 ターゲットユーザー
- 第1優先: 日本の大学生（学部1〜4年）
- 第2優先: 大学院生、専門学校生
- 第3優先: 社会人のレポート／研修課題

### 1.5 提供価値
1. **日本語学術文に特化**した自然な書き換え
2. **捏造しない安全性**（参考文献・固有名詞・数値を改変しない）
3. **だ・である調 / ですます調**の選択
4. **3秒〜15秒程度**で変換完了する軽快なUX

### 1.6 競合との差別化
| 項目 | Naoshi | Sidekicker等既存海外系 |
|---|---|---|
| 日本語学術文対応 | ◎（専用プロンプト） | △（多言語対応の一部） |
| 参考文献の保護 | ◎（明示的ルール） | × |
| 料金（週プラン） | ○（本物の週プラン） | ×（トライアル誘導） |
| だ／である調対応 | ◎ | × |

## 2. 課題と解決策

### 2.1 ユーザーの課題
1. AIで生成したレポートの「AIっぽさ」を教員に見抜かれる不安
2. 手動で書き換えると時間がかかる
3. 英語圏のヒューマナイザーは日本語学術文で不自然になる
4. 既存サービスは参考文献や固有名詞を勝手に改変してしまう

### 2.2 Naoshiの解決策
1. Claude Sonnet 4 + 日本語学術文特化のシステムプロンプト
2. 貼り付け→ワンクリックで変換
3. 日本語の文体・接続詞・語尾パターンを学習したプロンプト設計
4. 「捏造しない」「参考文献は保護する」ルールをプロンプトに厳守させる

## 3. 機能要件（MVPスコープ）

### 3.1 変換エンジン

#### 3.1.1 入力
- テキストエリア（プレーンテキスト貼り付け）
- 最大文字数: プランごとに制限（後述）
- 文体選択: `だ・である調`（デフォルト）/ `ですます調`

#### 3.1.2 処理
- 変換モード: **ノーマルモード（道A）のみ**
  - 捏造NG（事実・体験・数値・固有名詞の改変禁止）
  - 参考文献ブロックは完全保護
  - 文体・語彙・接続詞・語尾を自然化
- バックエンド: Claude Sonnet 4 API（Anthropic SDK）
- 変換ルールの実体: `prompts/humanize-system-prompt.md`（プロンプトのソース・オブ・トゥルース）
  - ※ プロジェクトルートの `CLAUDE.md` は Claude Code（開発補助AI）用の指示書なので別物。混同しないこと。

#### 3.1.3 出力
- 変換後テキスト（同じ画面に並列表示 or タブ切替）
- コピーボタン
- 文字数カウンタ

#### 3.1.4 非対象（MVP外）
- ディープモード（道B、ユーザーがコンテキスト提供して書き換え）
- AI検出スコア表示（GPTZero等連携）
- 盗作チェッカー
- レポート構成アドバイス
- 画像／PDF入力

### 3.2 認証・ユーザー管理
- Supabase Auth使用
- 認証方式:
  - Email + Magic Link（メイン）
  - Google OAuth（サブ、MVPで入れるかは要判断）
- 未ログインユーザー: サービス紹介のみ閲覧可、変換機能は利用不可
- プロフィール情報: email, 登録日, プラン, 累計利用数

### 3.3 料金プラン・課金
- Stripe Checkout + Customer Portal + Webhook
- プラン切り替え、解約、再開はCustomer Portalで完結
- 課金状態はSupabase DBにキャッシュし、Webhookで同期

（プラン詳細は第7章）

### 3.4 利用制限・カウント管理
- 無料プラン: **回数制限**（300字 × 3回/月）
- 有料プラン: **月間文字数制限**（ライト: 70,000字/月、ヘビー: 150,000字/月）
- 全プラン共通: 1回あたりの入力文字数上限あり（プランごとに異なる）
- リセット: 毎月1日 JST 0:00
- 超過時: モーダルで「今月の利用上限に達しました」と表示 → アップグレード導線
- カウンタは `usage` テーブルで記録（後述データモデル）
- 残り文字数（または残り回数）をUI上に常時表示

### 3.5 その他機能
- 変換履歴表示（最新10件、ログインユーザーのみ、MVPでは自分のみ閲覧）
- 問い合わせフォーム（Google Form埋め込みで代替可）
- 利用規約 / プライバシーポリシー / 特定商取引法に基づく表記（必須）

## 4. 非機能要件

### 4.1 性能
- 変換レスポンス目標: 3,000字で15秒以内（P95）
- 同時リクエスト: 初期は10同時想定

### 4.2 セキュリティ
- Row Level Security（RLS）で `usage` `subscription` テーブルを保護
- APIキー（Anthropic, Stripe）は`.env`管理、Vercel環境変数へ
- HTTPS必須
- 入力テキストはログに残さない（プライバシー保護、エラーログにも本文を出さない）

### 4.3 可用性
- Vercelホスティング（SLA準拠）
- Supabase: 無料プラン開始→有料移行を必要に応じて
- Claude APIがダウンした場合はユーザーにエラー表示＋後で再試行を促す

### 4.4 コンプライアンス
- 日本の特定商取引法に基づく表記を掲載
- 返金ポリシーを明記（デジタルサービスにつき基本返金不可、ただし未使用分は個別対応）

## 6. プロンプト設計方針（`prompts/humanize-system-prompt.md` 連携）

### 6.1 原則
1. **捏造禁止**: 事実・体験・数値・固有名詞・参考文献の改変は一切行わない
2. **参考文献の保護**: 著者名・書名・年号・直接引用は原文のまま
3. **文体の自然化**: AI特有の定型表現を避ける
4. **文脈的な接続詞**: 「1つ目は／2つ目は」のような機械的列挙を避け、文脈に応じた接続を選ぶ
5. **語尾の多様化**: 「〜ている／〜ない」の連続を避け、だ・である調では30〜40%を「〜だ／である」で締める
6. **冒頭の多様化**: 「本レポートでは」系の定型を避け、10パターン以上から選択
7. **短長対応**: 短文（〜800字）と長文（800字〜）で書き換え戦略を変える
   - 短文: 語尾・語彙の差し替えメイン
   - 長文: 段落構造の軽い組み替え＋能動的引用の活用
8. **能動的引用**: 参考文献がある場合は「佐藤(2020)が指摘するように」のような形で引用を自然化

### 6.2 `prompts/humanize-system-prompt.md` の役割
- 変換エンジンのシステムプロンプトの真実の情報源（source of truth）
- `/api/humanize` ルートがファイルを読み込み、Claude API のシステムプロンプトとして渡す
- バージョン管理し、変更履歴を残す
- 文体テスト結果に応じて継続的に改善
- **注**: プロジェクトルートの `CLAUDE.md` と `AGENTS.md` は Claude Code（開発補助AI）用のファイルであり、本ファイルとは役割が異なる。

### 6.3 文体オプション
- `だ・である調`（学術レポート向け、デフォルト）
- `ですます調`（一般的なレポート／感想文向け）

## 7. 料金プラン詳細

| プラン | 料金 | 1回あたり上限 | 月間制限 | 備考 |
|---|---|---|---|---|
| **無料** | ¥0 | 300字 | 3回/月 | メール登録必須、毎月1日リセット（JST） |
| **ライト（週）** | ¥500 / 週 | 2,000字 | 70,000字/月 | 本物の週プラン、自動更新、いつでも解約可 |
| **ヘビー（月）** | ¥2,980 / 月 | 3,000字 | 150,000字/月 | 月額、自動更新 |
| **ヘビー（年）** | ¥24,000 / 年（実質¥2,000/月） | 3,000字 | 150,000字/月 | 年額一括、月額換算で33%OFF |

### 7.1 プラン変更ルール
- アップグレード: 即時反映、差額日割り
- ダウングレード: 次回更新日から反映
- 解約: Customer Portalから即時可、期間末まで利用可

### 7.2 超過時の対応
- モーダルで「今月の利用上限に達しました。プランをアップグレードすると引き続きご利用いただけます。」と表示
- 「プランを変更する」ボタンでアップグレード導線
- 追加パック購入（従量課金）はMVPでは非対応（将来機能として `future-features.md` に記載）

### 7.3 リセットタイミング
- 全プラン共通: **毎月1日 JST 0:00** にリセット
- ライト（週プラン）も月間文字数ベースのため、月1日リセットで統一
- 月途中にプランをアップグレードした場合: 即時にアップグレード後の上限が適用される

### 7.4 API原価試算（目安）
- Claude Sonnet 4: input $3 / MTok, output $15 / MTok
- 3,000字（約6,000トークン）の変換: 入力+出力で約¥10〜¥20/回
- 各プランの原価と利益率:

| プラン | 最大原価/月 | 売上/月 | 利益率 |
|---|---|---|---|
| 無料 | 約¥50 | ¥0 | — |
| ライト | 約¥700 | 約¥2,000 | 65% |
| ヘビー | 約¥1,500 | ¥2,980 | 50% |

※ 全ユーザーが上限まで使い切ることは稀なため、実際の利益率はさらに高い

### 11.0 品質検証ステップ（M4〜M5の合格基準）

変換エンジンの品質を定量的に検証するため、以下のAI検出通過テストを必ず実施する。これはサービス内蔵機能ではなく、**開発時の品質検証ツール**として使う。

#### 11.0.1 M4: AI検出通過テスト（内部検証）

**テストサンプル**
- ChatGPT生成の日本語レポート × 5本
- Claude生成の日本語レポート × 5本
- Gemini生成の日本語レポート × 5本
- 各生成AIで「短文（〜800字）」「中文（800〜2000字）」「長文（2000字〜）」を網羅
- 合計15本

**検証フロー**
1. 上記15本をNaoshiで変換（ノーマルモード、だ・である調／ですます調の両方）
2. 変換後のテキストを以下の無料AI検出サービスに投入：
   - **GPTZero**（大学教員の利用率が高い定番）
   - **Copyleaks**（AI検出＋盗作チェック）
   - **ZeroGPT**（手軽な無料ツール）
3. 各サンプルのAI確率スコアを記録

**合格基準**
- **「Human-written」判定または「AI確率40%以下」のサンプルが80%以上（15本中12本以上）**
- この基準を満たさない場合、`prompts/humanize-system-prompt.md` のチューニングを繰り返す

#### 11.0.2 M5: Turnitin実地テスト（ベータ検証）

Turnitinは日本の大学でも導入が進んでおり、**実運用における最大の敵**。以下を実施する。

1. 大学に所属する協力者（知人）経由でTurnitinにサンプルを投入
2. 変換前後のAI検出スコアを比較
3. スコアが高く出たサンプルをフィードバックとしてプロンプトチューニングに反映
4. 合格基準: Turnitinで「AI確率30%以下」を70%以上のサンプルで達成

#### 11.0.3 注意事項
- AI検出サービスは日々アップデートされるため、定期的（月1回程度）に再検証する運用を前提とする
- 検出通過を目的化しすぎて「捏造NG」等の上位ルール（`prompts/humanize-system-prompt.md` 第1章）を破らないこと
- テスト結果は `naoshi/test-results/` などに記録して履歴を残す（プライバシーに配慮し、入力テキストは匿名化）

### 11.1 エージェント並行開発構成（VS Code + Claude Code想定）

M1〜M3の実装フェーズでは、Claude Codeのサブエージェント機能を使い、以下4エージェントで並行開発する。

| エージェント | 担当領域 | 前提依存 |
|---|---|---|
| **A: 基盤構築** | Next.js 14+セットアップ、Tailwind設定（design.md反映）、shadcn/ui導入、ディレクトリ構造、`.env.local.example` | なし |
| **B: UI/LP** | `/`（LP）、`/pricing`、`/login`、共通レイアウト、フッター、法的ページ | Aの完了後 |
| **C: 変換エンジン** | `/app`画面、`/api/humanize`ルート、Claude SDK連携、`prompts/humanize-system-prompt.md`読み込み、文字数カウンタ、コピー機能 | Aの完了後 |
| **D: 認証・課金** | Supabase Auth統合、Stripe Checkout/Portal/Webhook、`/account`、`profiles`/`subscriptions`/`usage`テーブル、利用制限ミドルウェア | Aの完了後 |

#### 並行実行ルール
1. **A（基盤構築）は単独で先行**。完了後にB/C/Dを並列起動。
2. B/C/Dはそれぞれ独立したディレクトリ／ブランチで作業し、コンフリクトを最小化。
3. 各エージェントは担当範囲外のファイルを変更しない。
4. 最後に統合レビュー＆手動結合テスト（M4フェーズ）。

#### 各エージェントへの共通ブリーフ
- `requirements.md` を読んで機能要件を把握
- `design.md` を読んでUI仕様を把握
- `prompts/humanize-system-prompt.md` はCエージェントのみが参照（変換エンジンのシステムプロンプト）
- コードコメントは日本語
- 機密情報はコミットしない
